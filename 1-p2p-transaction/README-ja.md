# システムデザインドキュメント: P2P送金システム

## 1. サマリ（Executive Summary）

### 1.1 システム概要
本システムは、Zelle/Venmoライクなリアルタイムのピアツーピア（P2P）送金システムです。ユーザーは電話番号またはメールアドレスを指定するだけで、即座に他のユーザーへ送金できます。

### 1.2 機能要件
| 要件 | 内容 |
|------|------|
| コア機能 | 送金、受取、残高照会、取引履歴表示 |
| 識別方法 | 電話番号またはメールアドレス |
| 通貨 | USD単一通貨のみ |
| 送金制限 | 1回あたり$500、1日あたり$2,500 |
| 決済方式 | 即時決済（ACH方式ではない） |
| 対象外 | アカウント作成、投資機能、多通貨対応 |

### 1.3 非機能要件
| 要件 | 目標値 | 備考 |
|------|--------|------|
| ユーザー規模 | 登録ユーザー5,000万人、DAU 500万 | |
| スループット | 平均200 TPS、ピーク1,000 TPS | |
| レイテンシ | エンドツーエンドで3秒以内 | 送金完了まで |
| 可用性 | 99.99% | 年間ダウンタイム約50分 |
| データ保持 | 取引履歴を7年間保持 | 規制要件 |
| 地域 | 米国のみ（初期ローンチ） | |

### 1.4 セキュリティ・コンプライアンス
- **KYC（本人確認）**: 上流サービスで実施済み（スコープ外）
- **認証**: 既存の認証サービスを利用（スコープ外）
- **不正検知**: ルールベースの不正チェックをインライン実装、ML-basedスコアリングは外部API
- **AML（マネーロンダリング対策）**: 外部APIコール
- **コンプライアンス**: SOC2準拠（PCI-DSSは対象外）

### 1.5 技術スタック前提
- **クラウドプロバイダー**: AWS
- **アーキテクチャ**: マイクロサービスアーキテクチャ（理由は後述）
- **スコープ**: バックエンドとAPIコントラクトに集中（フロントエンド/モバイルは対象外）

### 1.6 設計方針（Design Principles）
本ドキュメントは「概要 → 設計方針 → 詳細」の順で段階的に詳細化する。金融ドメインであるため、以下を最優先の設計原則とする。

1. **正確性 (Correctness) を最優先**: お金は失っても増やしてもいけない。残高は常に整合し、二重支払いは構造的に防止する。
2. **冪等性 (Idempotency)**: ネットワークリトライによる二重送金を、Idempotency-Key で確実に防ぐ。
3. **不変な監査ログ (Immutable Audit Log)**: 台帳は追記専用（append-only）。残高は仕訳の集約結果であり、上書きしない。
4. **段階的整合性の使い分け**: 残高・台帳は**強整合（同期）**、通知・履歴反映は**結果整合（非同期）**。
5. **障害前提の設計**: あらゆるコンポーネントが落ちうる前提で、SPOFを排除しMulti-AZで冗長化する。
6. **同期で減算、非同期で決済**: 送金者はエスクローへ同期で減算（即時・確定的なフィードバック＋資金確保）し、受取人への加算は非同期ワークフロー（fraud/AML → モラトリアム → 決済）で行う。「即時決済」要件は**リスク階層別のモラトリアム**で満たす — 低リスクの送金は既定で約0（実質即時）、高リスクのみ短時間ホールドする。§4.3 と §7.2 のトレードオフ参照。

#### 容量見積もり（Back-of-the-envelope）
| 項目 | 計算 | 結果 |
|------|------|------|
| 平均書き込み | 200 TPS | 1送金あたり台帳2行 → 400 行/s |
| ピーク書き込み | 1,000 TPS | 2,000 行/s（バースト） |
| 読み取り（残高/履歴） | 書込の約10倍 | 〜2,000〜10,000 RPS |
| **台帳の行数/年** | 200 TPS × 2行 × 86,400 × 365 | **約126億行/年**（7年で約880億行） |
| 取引データ量/年 | 約126億行 × 約1KB | 約 13 TB/年（7年で約 90 TB） |
| 必要な残高更新精度 | 整数（minor unit = セント） | 浮動小数点は使わない |

> **重要**: 金額はすべて `bigint`（セント等のminor unit）で保持し、丸め誤差を排除する。

> **重要なストレージ判断 — ポリグロット永続化、金額はDynamoDB**: 約126億行/年はPostgreSQLには大きすぎるため、トランザクション/台帳系と残高はすべて**DynamoDB**に置く（この量を水平スケールできる）。PostgreSQLは**金額以外のメタデータ**と残高の**表示キャッシュ**のみを持つ:
> - **DynamoDB（金額の正）** — 全 `ledger_entries`（追記専用、7年、`account_id` でパーティション、ソートキー `seq`）。**残高用の別レコードは持たない: 各台帳エントリが処理後の `balance_after` を記録するので、現在残高は最新エントリの `balance_after` そのもの。** 資金移動は均衡したエントリを追記する原子的な **`TransactWriteItems`**。二重処理はクライアントの **Idempotency-Key**（条件付き `attribute_not_exists`）で防ぐ。
> - **PostgreSQL（メタデータ + キャッシュ）** — `users`、`user_identifiers`、`transfers` のメタ（status、検索可能なメモ）、`transfer_events`（ワークフロー状態）、`idempotency_keys`。加えて **DynamoDB Streams から更新される残高の表示キャッシュ**（`balance_cache_minor`）— *決して*正ではなく、*決して*減算判定に使わない。

## 2. ハイレベルアーキテクチャ（High-Level Architecture）

### 2.1 コンポーネントベースのアーキテクチャ

![Architecture Overview](./architecture-overview-ja.png)

リクエストは `CloudFront → WAF → API Gateway/ALB` のエッジ層を通り、JWT検証後に適切なサービスへルーティングされる。送金フロー・二重仕訳台帳・履歴照会はすべて単一の **Payments Service** 内で強整合トランザクションとして処理される。通知・受取人解決・不正検知は独立サービスへ委譲し、通知/履歴のファンアウトはイベントバス経由で非同期化する。

#### サービス分割の方針
URLパスでサービスを分割する設計（例: `/transfers`、`/transactions`、`/accounts` ごとに1サービス）は**採らない**。サービスはAPI表面ではなく、**トランザクション境界とデータ所有権**で分割する。

- 送金フロー・残高更新・二重仕訳台帳・取引履歴は、いずれも**同一データ**を読み書きし、**同一のACIDトランザクション**に参加する必要がある（例: debit/credit と `transfers` のステータス更新は原子的でなければならない）。これらを分割すると、コアの資金移動パスに分散トランザクション/Sagaを強いることになり、メリットなく複雑性だけが増す。→ よって単一の **Payments Service** に統合する。
- **Fraud Engine**、**User Directory**、**Notification** は本質的に独立した関心事（データが異なり、障害モードが異なり、スケール特性が異なり、単独で呼び出せる）→ よって**独立サービスとして存続**させる。

#### この分割の理由（vs モノリス / vs URLベース分割）
| 観点 | 判断 |
|------|------|
| **資金移動を単一トランザクションで** | 送金 + 台帳 + 履歴はデータを共有し原子的でなければならない。分散Sagaではなく単一DBトランザクションを使うため、単一の **Payments Service** にまとめる |
| **障害分離** | Notification / Fraud を分離し、それらの障害が資金移動コアをブロックしないようにする（グレースフルな劣化） |
| **必要な箇所での独立スケール** | 読み取りの多い履歴は Payments Service *内*のリードレプリカで処理。Fraud/Notification はそれぞれの負荷特性でスケール |
| **トレードオフ** | Payments Service は大きくなる（読み書き両方を所有）が、クリティカルパスの分散トランザクションの複雑性を回避できる。読み取りスケールは別サービス化ではなくCQRSのリードレプリカで対処 |

> **段階的アプローチ**: コアとなる **Payments Service** と、独立した **Fraud / User Directory / Notification** から始める。実際のトランザクション/スケール境界が要求したときのみさらに分割する。単にURLプレフィックスが異なるという理由だけで分割しない。

#### 主要コンポーネントの責務
| コンポーネント | 責務 |
|----------------|------|
| **API Gateway / ALB** | JWT検証、レート制限、ルーティング、TLS終端 |
| **Payments Service** | 資金移動のコア。送金フローのオーケストレーション、冪等性制御、限度額チェック、不正検知の呼び出し。残高の強整合更新と二重仕訳台帳への追記（**唯一の残高更新主体**）。CQRSの読み取りで取引履歴を提供（7年保持）。資金移動の状態はすべて単一のトランザクション境界内に存在する |
| **User Directory Service** | 電話番号/Email → `user_id` の解決（受取人特定） |
| **Notification Service** | Push/SMS/Email通知（非同期、結果整合） |
| **Fraud Rule Engine** | インラインのルールベース不正チェック。**外部リスクAPIとの連携をすべて所有**し、Payments Service に代わって ML不正スコアリングAPI と AMLスクリーニングAPI を呼び出す |
| **Event Bus (Kafka/SQS)** | サービス間の非同期連携。アウトボックスからのイベント配信 |

#### 外部連携
外部のリスクAPIは **Payments Service が直接呼び出すことはなく**、リスク/コンプライアンスの単一の連携点である **Fraud Service** を経由してアクセスする。これにより、不正/AMLの関心事（ベンダーSDK、認証情報、フォールバック方針、false positiveの扱い）を資金移動のコアから切り離す。

- **ML不正スコアリング API**: Fraud Service が呼び出す。高リスク判定時のみ同期呼び出し（タイムアウトでフォールバック）
- **AML スクリーニング API**: Fraud Service が呼び出し、制裁リスト照合を行う
- **銀行/決済ネットワーク（RTP / FedNow）**: 即時グロス決済。本設計では「資金移動の最終ネットワーク」として外部化

### 2.2 Kubernetes（EKS）ベースのデプロイメント

![Kubernetes Architecture](./architecture-k8s-ja.png)

AWS上で **Amazon EKS** にコアサービスをデプロイする。データストア（Aurora / ElastiCache / MSK）はマネージドサービスとしてクラスタ外に置き、運用負荷とブラスト半径を下げる。

| 項目 | 採用 | 理由 |
|------|------|------|
| **オーケストレーション** | Amazon EKS（Multi-AZ） | ノード障害・AZ障害への自動復旧 |
| **オートスケール** | HPA（CPU/カスタムメトリクスTPS）+ Cluster Autoscaler | ピーク1,000 TPSへの追従 |
| **サービスメッシュ** | Istio / App Mesh | mTLS、リトライ、サーキットブレーカ、可観測性 |
| **シークレット管理** | External Secrets Operator → Secrets Manager | DB認証情報をコードから分離 |
| **デプロイ戦略** | ローリング / カナリア（Argo Rollouts） | ゼロダウンタイム更新（99.99%目標） |
| **Pod配置** | PodAntiAffinity + topologySpreadConstraints | 同一AZ集中を防ぎSPOF排除 |

## 3. 共通コンポーネント（Cross-Cutting Concerns）

| 領域 | 採用技術 | 内容 |
|------|----------|------|
| **認証 / 認可** | 上流の認証サービス（JWT） + API Gatewayで検証 | KYC済前提。サービス間は **mTLS**。リソースアクセスはJWTクレーム（sub=userId）で認可 |
| **メトリクス監視** | Prometheus + Grafana / CloudWatch | TPS、レイテンシ（p50/p95/p99）、エラー率、残高更新失敗率 |
| **分散トレーシング** | OpenTelemetry + AWS X-Ray | `traceId` を全サービスに伝播。送金フローのボトルネック特定 |
| **ログ** | Fluent Bit → OpenSearch / S3 | 構造化ログ（JSON）。PII はマスキング。監査ログは別系統で改ざん防止 |
| **アラート** | Alertmanager / PagerDuty | SLO違反（可用性/レイテンシ）、残高不整合検知、不正スパイク |
| **監査ログ** | 追記専用ストア + S3 Object Lock（WORM） | 7年保持。SOC2要件。改ざん不可 |
| **設定管理** | AWS AppConfig / ConfigMap | 限度額やフラグを無停止で変更 |
| **シークレット** | AWS Secrets Manager + 自動ローテーション | DB/外部API認証情報 |
| **レート制限** | API Gateway + Redis（トークンバケット） | ユーザー単位・IP単位 |

#### 主要SLI/SLO
| SLI | SLO |
|-----|-----|
| 送金成功率 | ≥ 99.95% |
| 送金レイテンシ p99 | ≤ 3秒 |
| 可用性 | 99.99%（年間約50分） |
| 残高整合性 | 100%（日次リコンサイルで検証） |

## 4. アプリケーション設計概要（Application Design）

### 4.1 データモデル（エンティティ一覧）

![Database Schema](./database-schema-ja.png)

| エンティティ | ストア | 役割 | 重要ポイント |
|--------------|--------|------|--------------|
| **台帳** | **DynamoDB** | 全二重仕訳台帳 **+ 残高** | 全エントリ、追記専用、7年。パーティションキー `account_id`、ソートキー `seq`（勘定ごとの連番）、`transfer_id` にGSI。各エントリが `balance_after` を持つので、**現在残高は最新エントリの `balance_after`** — 別の残高レコードは不要。約126億行/年。**金額 + 履歴・監査**の正 |
| **users** | PostgreSQL | ユーザー（KYC済） | 上流から同期。参照中心のメタデータ |
| **user_identifiers** | PostgreSQL | Email/電話 → userId | `value` にユニーク索引。受取人解決に使用 |
| **accounts (メタ)** | PostgreSQL | 勘定メタ + 残高**キャッシュ** | `type`（USER / **ESCROW**）、通貨など。`balance_cache_minor` は**表示キャッシュのみ**（DynamoDB Streamsから更新）、正ではなく減算判定に使わない |
| **transfers (メタ)** | PostgreSQL | 送金メタ/検索/ワークフロー | ライフサイクル `status`（`DEBITED → FRAUD_REVIEW → MORATORIUM → SETTLED`、または `CANCELLED`/`REVERSED`）、検索可能な `note`、`settle_after`。`idempotency_key` にユニーク制約。**金額はDynamoDBにあり、ここにはない** |
| **transfer_events** | PostgreSQL | ワークフロー追跡 | 各非同期ステップ（`FRAUD`/`MORATORIUM`/`SETTLE`/`NOTIFY`）とその状態 — 決済ワークフローの永続状態 |
| **idempotency_keys** | PostgreSQL | 冪等性制御 | リクエストハッシュ + レスポンス。（DynamoDBトランザクション自身も冪等性ガードアイテムを持つ — 後述） |

#### 二重仕訳（Double-Entry Ledger）の考え方（エスクロー併用）
受取人への加算は**遅延**する（fraud＋モラトリアム後の非同期決済）ため、その間にお金が消えないよう、同期ステップでは送金者から**システムのエスクロー（clearing）勘定**へ資金を移し、決済時にエスクローから受取人へ移す。各phaseは**合計ゼロの仕訳ペア**になる。

各phaseは1回の **DynamoDB `TransactWriteItems`** で、均衡した台帳エントリを追記する。各エントリは直前の最新エントリから計算した `balance_after` を記録する — つまり**残高は台帳自身が持つ**、別アイテムではない:

```
Phase 1 (同期, POST時):   送金者 → エスクロー   [1回の TransactWriteItems]
  送金者の最新エントリ(balance_after=B_s, seq=N_s)とエスクロー(B_e, N_e)を読む
  - 送金者エントリput (seq=N_s+1, amount=-1000, balance_after=B_s-1000, phase=DEBIT_TO_ESCROW)
        条件: (送金者, N_s+1) のアイテムが存在しない   ← 最新エントリへの楽観ロック
  - エスクローエントリput (seq=N_e+1, amount=+1000, balance_after=B_e+1000, phase=DEBIT_TO_ESCROW)
        条件: idempotency_key が未記録               ← 二重処理を防止

Phase 2 (非同期, 決済時):  エスクロー → 受取人   [1回の TransactWriteItems]
  - エスクロー + 受取人エントリput (phase=ESCROW_TO_RECIPIENT)、同じ条件

  → 各phaseで Σ amount = 0 （資金の保存則。決済後はエスクローがゼロに戻る）
  取消時は REVERSAL の TransactWriteItems（エスクロー → 送金者）を書く。
```

**真実はどこにあるか**: **金額はDynamoDBの台帳にある。** 各台帳エントリは1トランザクションで処理後の `balance_after` を記録する。現在残高は**最新エントリの `balance_after`**（勘定パーティションの最大 `seq`）を直接読むだけ — 同期すべき別の残高レコードはない。正しさは追記時の2つの条件から来る: `seq=N+1` 未存在の条件は**楽観ロック**で、並行する2書込が同じ最新エントリから連鎖できない（負けた方は新しい最新に対して再試行）。**Idempotency-Key** の条件は再送リクエストをno-opにする（二重減算なし）。**PostgreSQLは正の金額を一切持たない** — `balance_cache_minor` は DynamoDB Streams から非同期更新される表示キャッシュで、減算判定には決して読まない。これは意図的な**ポリグロット永続化**: 巨大で水平スケールする台帳はDynamoDB、リッチなリレーショナルなメタ・検索・ワークフロー状態はPostgreSQL。

### 4.3 送金フロー（シーケンス）

![Transfer Sequence](./transfer-sequence-ja.png)

フローは意図的に、**短い同期パス**（`POST /v1/transfers` を高速に返す）と**非同期の決済ワークフロー**に分割している。

**同期（`POST /v1/transfers` → 200 OK + イベント）:**
1. **メタ + 冪等性**: PostgreSQLに `transfers` メタ行を書く（`idempotency_key` にユニーク制約）。重複は競合し保存済みレスポンスを返す。
2. **限度額チェック**（例: 1回$500、1日$2,500）と**受取人解決**をここで行う — 早期に安価に弾く。
3. **減算のみ、エスクローへ — DynamoDBで**: 送金者とエスクローの最新台帳エントリ（`balance_after`, `seq`）を読み、1回の **`TransactWriteItems`** で送金者エントリ（`seq+1`, `balance_after-=金額`）とエスクローエントリ（`seq+1`, `balance_after+=金額`）を追記する（共に `phase=DEBIT_TO_ESCROW`）。条件: `seq+1` が未存在（楽観ロック。`balance_after` が負になる場合も失敗＝残高不足）かつ Idempotency-Key が未使用。`ConditionalCheckFailed` は新しい最新に対して再試行、または再送の重複排除。受取人への加算は**まだ行わない**。
4. `200 OK (status=DEBITED)` を返し、`TransferRequested` をKafkaへ発行する。

**非同期の決済ワークフロー（ステートマシン。複雑化すれば AWS Step Functions）:**
1. **Fraud / AML チェック**: 怪しいマーチャント・AMLスクリーニング。ごく一部は**人による承認フロー**が必要になりうる。却下時は補償の `REVERSAL` `TransactWriteItems`（エスクロー → 送金者）を書き、`transfers.status=CANCELLED` にする。
2. **モラトリアム**: `settle_after` まで待機し、ユーザーが送金を取消できるようにする（取消可能な期間）。
3. **決済（実際の振込）**: 1回の `TransactWriteItems` でエスクロー → 受取人へ移す（`phase=ESCROW_TO_RECIPIENT`）、PostgreSQLの `transfers.status=SETTLED` に更新。
4. **通知**: 送金者・受取人の両方に通知（Push/SMS/Email）。

> **なぜこの分割か**: 同期で減算することで送金者に即時の確定的フィードバックを返しつつ資金を確保（DynamoDBの条件付き書込で二重支払い防止）でき、加算を遅延させることでクライアントのリクエストを保持せずに fraud/AML と取消期間を提供できる。エスクローにより各ステップで台帳が常に均衡し、`transfer_events`（PostgreSQL）によりワークフローはクラッシュ後も永続・再開可能になる。

## 5. セキュリティ（Security）

| 観点 | 対策 |
|------|------|
| **認証** | 上流発行のJWTを API Gateway で検証。短い有効期限 + リフレッシュ。サービス間は mTLS |
| **認可** | JWTの `sub`(userId) と操作対象アカウントの所有者一致を強制。最小権限のIAM |
| **転送中の暗号化** | 全経路 TLS 1.2+。クラスタ内も mTLS |
| **保存時の暗号化** | Aurora/Redis/S3 を KMS で暗号化。PIIフィールドはアプリ層でも暗号化検討 |
| **PII保護** | ログ・トレースで電話/Email/金額をマスキング。最小限の保持 |
| **入力検証** | スキーマ検証、金額の上限/正値チェック、SQLはパラメータ化（ORMでインジェクション防止） |
| **レート制限 / WAF** | ユーザー・IP単位のトークンバケット。WAFでL7攻撃・BOTを遮断 |
| **不正・AML** | Fraud Service がインライン ルールチェック（速度・新規受取人・金額異常）を行い、外部ML/AML APIの唯一の呼び出し元となる。Payments Service は Fraud Service のみを呼び出す |
| **冪等 & リプレイ防止** | `Idempotency-Key` + リクエストハッシュ照合（同一キーで異なるbodyは拒否） |
| **監査** | 全状態変化を不変ログに記録（誰が・いつ・何を）。S3 Object Lock で WORM 化 |
| **コンプライアンス** | SOC2（PCI-DSSはカード非取扱のため対象外）。職務分掌・アクセスレビュー |
| **シークレット** | Secrets Manager + 自動ローテーション。コード/イメージに秘密を埋めない |

> **不正検知の同期/非同期の切り分け**: ルールベース（低レイテンシ）は同期でブロック。MLスコアリングは高リスク候補のみ同期、それ以外はイベント駆動で事後分析（false positiveで正常送金を止めない）。

## 6. 非機能要件の実現（Non-Functional / Performance）

### 6.1 レイテンシ（目標 p99 ≤ 3秒）
- **同期パスの最小化**: 同期リクエストは「冪等チェック → 限度額チェック → 受取人解決 → エスクローへ減算 → 200返却」のみ。Fraud/AML・モラトリアム・決済・通知はすべて非同期ワークフローに移し、ユーザー向けレイテンシを3秒未満に十分収める。
- **残高読取**: 正の残高は**最新台帳エントリの `balance_after`** — 勘定パーティションの最大 `seq` を1クエリで読む（必要時は強整合読取）。PostgreSQLの `balance_cache_minor` は高速なUI表示に使えるが、**減算判定は必ずDynamoDBを読み・条件付け**し、キャッシュは使わない。
- **コネクションプーリング**: PostgreSQL接続は RDS Proxy / PgBouncer で再利用。DynamoDBはSDKの接続再利用でHTTPアクセス。
- **ホットアカウント注記**: 各勘定の書込は `seq` で連鎖するため、極端にホットな単一勘定（人気マーチャント等）は自分の書込を直列化する。これは正しい（真の累計残高は本来逐次的）。そうした勘定をスケールするには §7.2 のサブアカウント案を参照。

### 6.2 スループット & スケーラビリティ（200→1,000 TPS、将来10x）
- **ステートレスなコアサービス** + HPA で水平スケール。状態はDB/Redis/Kafkaに外出し。
- **ポリグロットストレージ**: 金額（残高を内包する台帳）はすべて **DynamoDB**。約126億行/年の書込・読込量を水平スケールでき、残高・履歴読取もここから。PostgreSQLは有界なメタ/検索/ワークフローのみを担い、リレーショナル層が巨大ストリームを背負わない。
- **ネイティブパーティショニング**: DynamoDBは台帳を `account_id` でパーティション（ソートキー `seq`）し、勘定間で負荷を分散。PostgreSQLのメタは小さく、必要なら `account_id` で分割/シャーディング可能（後述の調整事項）。
- **イベント駆動の負荷平準化**: バースト時はキューがバッファとなり、下流（通知等）を保護。

### 6.3 可用性（99.99%）
- **Multi-AZ**: EKSノード、Aurora、ElastiCache、MSK をすべて複数AZへ。
- **SPOFの排除**: 単一インスタンス依存をなくす（後述の深掘りQ4参照）。
- **グレースフルな劣化**: 通知/履歴が落ちても**送金本体は成立**する（イベントはアウトボックスに残り後で配信）。
- **サーキットブレーカ & タイムアウト**: 外部API（ML/AML/銀行）の遅延が全体を巻き込まないよう遮断＋フォールバック。

### 6.4 整合性 & 信頼性
- **DynamoDBでの原子的な資金移動**: 均衡した台帳エントリを1回の **`TransactWriteItems`** で条件付きに追記する。次の `seq` 未存在の条件がロスト更新（過剰支払い）を防ぎ、Idempotency-Key の `attribute_not_exists` が二重処理を防ぐ。`balance_after` は同じ書込内で計算。all-or-nothing — 部分的な減算/加算なし、過剰支払いなし、二重適用なし。
- **キャッシュは非同期・非正**: PostgreSQLの `balance_cache_minor` は **DynamoDB Streams** から更新（結果整合）。表示専用で、減算はDynamoDBを条件付きで読むため、キャッシュの古さが誤った減算を引き起こすことはない。
- **クロスストアのメタ**: `transfers` メタ書込（PostgreSQL）と金額書込（DynamoDB）は2ストア。メタ行（冪等キー）を先に書き、次に条件付きDynamoDBトランザクション、という順序にする。間のクラッシュはワークフローでリコンサイル（台帳エントリのない `DEBITED` メタ行は再試行またはロールバック）。
- **全所で冪等**: DynamoDBトランザクションのIdempotency-Key条件とKafka consumerの `transfer_id` 重複排除で再送は安全。
- **継続的リコンサイル**: 勘定の台帳エントリを再生し、各 `balance_after` が正しく連鎖（前 + amount = 次）すること、エスクローが処理中合計に一致することを検証するジョブを実行し、乖離はアラート。

## 7. 調整事項（Trade-offs & Tunable Decisions）

要件次第で変更しうるポイントと、そのトレードオフを示す。「正解は一つでない」ことを明示する。

### 7.1 ポリグロット永続化: 金額はDynamoDB、メタデータはPostgreSQL
データ量（約126億行/年）から台帳はPostgreSQLでは不可能なので、金額（残高を内包する台帳）は **DynamoDB** に置き、PostgreSQLはメタデータ + 表示キャッシュのみを持つ:

| 観点 | DynamoDB（金額: 台帳 + 残高） | PostgreSQL（メタデータ + キャッシュ） |
|------|-------------------------------|----------------------------------------|
| **保持** | 全台帳、7年、追記専用。残高 = 最新エントリの `balance_after` | users、transfersメタ、ワークフローイベント、**残高の表示キャッシュ** |
| **トランザクション** | `TransactWriteItems`（最大25項目）— 均衡エントリ + 条件を原子的 | メタデータの多行ACID（金額ではない） |
| **スケール** | `account_id` でパーティションし、ほぼ無限に水平スケール | 有界で小さなリレーショナルデータ。運用容易 |
| **整合性** | 減算判定は最新エントリの強整合読取 + 条件付き追記 | キャッシュは結果整合（DynamoDB Streams）。正ではない |
| **理由** | 巨大な書込スループット + 台帳の安価な長期保持 | リッチなクエリ: メモ検索、結合、ワークフロー状態、レポート |

> トレードオフ: 金額をDynamoDBに置くと、**過剰支払い防止と順序を条件付き追記（次の `seq` への楽観ロック）として表現**する必要があり、SQL `SERIALIZABLE` には頼れない。また勘定ごとの書込は `seq` 連鎖で直列化する。台帳量がPostgreSQLには非現実的で、条件付き `TransactWriteItems` がスケール（勘定間）と正確性（勘定内）の両方を与えるため、これを受け入れる。PostgreSQLのメタ/キャッシュはDynamoDBと結果整合（Streams経由）で、継続ジョブでリコンサイル。**代替案**: (a) 全PostgreSQL + パーティショニング + S3アーカイブ — 整合は最も単純だが行数が運用上非現実的、(b) 金額はPostgreSQL・履歴はDynamoDB（以前のイテレーション）— 残高にSQL ACIDを保てるがクロスストア同期が依然必要で書込スケールがリレーショナルWriterで頭打ち。台帳スケールが支配的なときは台帳全体をDynamoDBに置くのが選択。

### 7.2 その他の調整ポイント
| 調整軸 | 選択肢A | 選択肢B | トレードオフ |
|--------|---------|---------|--------------|
| **送金確認** | 完全同期（即時COMPLETED） | **同期減算 + 非同期決済**（採用） | 完全同期はUXが最もシンプルだが、fraudレビュー/取消の余地がなく、決済がリクエストの可用性に結合する。本設計は「エスクローへ同期減算 + 非同期決済」を採用: 送金者は即時の確定的フィードバックを得つつ、fraud/AMLとモラトリアムをクリティカルパス外で実行。トレードオフ: 受取人は即時ではなく決済後に着金 — モラトリアム/fraud要件を踏まえれば許容 |
| **モラトリアム長** | 0（即時決済） | 数分〜数時間 | 長いほど取消/fraudの安全性は上がるが体感の着金は遅くなる。`settle_after` でリスク階層（新規受取人・高額等）ごとに調整可能 |
| **整合性モデル** | 強整合（DynamoDBの残高/台帳） | 結果整合（PG残高キャッシュ、通知） | 金額はDynamoDBの強整合な条件付き書込を使う。PostgreSQLの表示キャッシュと通知は可用性のため結果整合（CAPのトレードオフ） |
| **極端にホットな勘定** | 勘定ごと単一台帳（採用） | N個の**サブアカウント**に分割 | 勘定ごとの書込は `seq` 連鎖で直列化するため、極端にホットな勘定（巨大マーチャント）は書込ホットスポットになりうる。必要ならN個のサブアカウント（各々が独自の台帳パーティション）にモデル化し、合算/ルーティングする — ルーティング+集約の複雑性が増すので、稀なホットスポットにのみ適用しデフォルトにはしない |
| **メッセージング** | Kafka(MSK) | SQS/SNS | Kafka=高スループット/再処理に強い、SQS=運用簡単。規模で選択 |
| **不正検知** | インラインのみ | インライン+ML | レイテンシ vs 検知精度。高リスクのみML同期で両立 |
| **マルチリージョン** | 単一リージョン（現行,US-only） | アクティブ/パッシブDR | コスト vs RTO/RPO。初期は単一+リージョンバックアップ |
| **決済ネットワーク** | クローズドループ（内部残高のみ） | RTP/FedNow連携 | 内部完結は即時・低コスト、外部連携は実銀行口座反映だが遅延/手数料 |

## 8. 想定問答集（Anticipated Q&A / Deep Dives）

要件 Part 3 の深掘り質問を中心に、面接で問われやすい論点と想定解答をまとめる。

### Q1. サーバが送金の途中でクラッシュしたら、どう復旧する？
**A.** 各資金移動ステップ（同期のエスクロー減算、非同期の決済）は単一の **DynamoDB `TransactWriteItems`** で、all-or-nothing — コミット前のクラッシュは部分的な減算/加算を残さない。決済ワークフローは**メモリ上ではなく永続的**で、進捗は `transfers.status` と `transfer_events`（PostgreSQL）に保持される。ワークフロー途中のクラッシュは永続化された最後の状態から再駆動して復旧する。各ステップは冪等 — DynamoDBトランザクションが `transfer_id`/phase をキーとするガードアイテムを持つため、再実行は二重適用ではなくno-op（`ConditionalCheckFailed`）になる。受取人への加算は決済時のみ（同期ステップでは行わない）なので、減算後のクラッシュでは資金は**エスクロー**に安全に留まり、ワークフロー再開を待つ。唯一のクロスストアの隙間 — DynamoDBの金額書込の直前に書くPostgreSQLの `transfers` メタ — はリコンサイルする: DynamoDBに対応する台帳エントリのない `DEBITED` メタ行はスイーパが再試行/ロールバックする。複雑化すれば AWS Step Functions がこの永続実行を標準提供する。

### Q2. ネットワークリトライで同じリクエストが2回届いたら、二重送金をどう防ぐ？
**A.** クライアントが生成する `Idempotency-Key` を必須化する。2層: (1) PostgreSQLの `transfers` メタ行が `idempotency_key` に**ユニーク制約**を持ち、重複作成は競合して前回のレスポンスを返す。(2) より重要なのは、金額更新の **`TransactWriteItems` がガードアイテムを含む**こと — `transfer_id`/phase をキーに `attribute_not_exists` 条件を付けるので、同一トランザクションを再試行しても条件付き書込が失敗（`ConditionalCheckFailed`）し、残高はちょうど1回だけ減算される。冪等性はメタ層だけでなく、DynamoDB内で金額書込を原子的に保護する。保存したリクエストハッシュにより「同一キーで異なるbody」も矛盾として拒否できる。

### Q3. 残高の読み取りと更新が並行したら、整合性をどう担保する？
**A.** 残高は台帳が持つ: 各新エントリは勘定の現在の最新エントリ（`seq=N`, `balance_after=B`）から計算した `balance_after` を記録する。追記は次のエントリを `seq=N+1` に書き、その条件として **`seq=N+1` のアイテムが存在しない**ことを付ける（楽観ロック）。並行する2つの減算は両方 `seq=N` を読み `seq=N+1` への書込を試み、DynamoDBはちょうど一方を勝たせ、他方は `ConditionalCheckFailed` で新しい最新（`seq=N+1`、`balance_after` を再計算）に対して再試行する。よってロスト更新も過剰支払いも起きない — 読み・計算・追記が勘定ごとに事実上直列化され、チェックと書込が1つの原子的な条件付き操作（TOCTOUなし）。この判定でPostgreSQLキャッシュは**決して**読まない。リコンサイルジョブが連鎖を再生して各 `balance_after` の整合を確認する。

### Q4. SPOF（単一障害点）はどこにあり、どう排除する？
**A.**
- **金額ストア**: DynamoDBは設計上マルチAZ・リージョン内冗長（失う単一ノードがない）。正の残高/台帳に単一障害点がない。
- **メタDB**: Aurora Multi-AZ（自動フェイルオーバ）＋リードレプリカ。
- **キャッシュ**: ElastiCache クラスタモード（シャード＋レプリカ）。加えてPostgreSQLの残高キャッシュは非正なので、その古さ/障害が減算（DynamoDBを読む）をブロックすることはない。
- **メッセージング**: MSK を複数ブローカ/AZへ。
- **コンピュート**: EKSをMulti-AZ、PodAntiAffinityでAZ分散、HPAで冗長。
- **エッジ**: CloudFront/Route53でDNSフェイルオーバ。
- **外部API**: サーキットブレーカ＋タイムアウト＋フォールバック（例: MLがダウンならルールベース判定にデグレード）。

### Q5. トランザクション量が10倍（〜10,000 TPS）になったら、ボトルネックはどこで、どうスケールする？
**A.** 台帳は**DynamoDB**にあり `account_id` でパーティションされるので、総スループットはほぼ線形に水平スケールし、金額ストアが最初の壁ではない。圧力点と対策は段階的に、
1. **極端にホットな単一勘定**が主リスク — 自分の書込が勘定ごとの `seq` 連鎖で直列化する（正しいがホットスポット）。1勘定が支配的なら、N個の**サブアカウント**（各々が独自の台帳パーティション）にモデル化し、ルーティング/集約する（§7.2参照）。大半の勘定はこれを必要としない。
2. 同期パスは最小（1回の `TransactWriteItems` でエスクローへ減算）に保ち、fraud/モラトリアム/決済/通知は非同期ワークフローへ。
3. **DynamoDBオンデマンド/オートスケール**で容量を書込ストリームに追従させる。`account_id` パーティションキーが負荷を分散し、総体としてホットパーティションを避ける。
4. **PostgreSQLメタ**は台帳に比べ有界で小さい。リードレプリカでスケールし、必要時のみ `account_id` で分割。残高キャッシュはDynamoDB Streamsのconsumerが更新する。
5. メッセージングはKafkaパーティション増設、ワークフローは永続状態に駆動されるステートレスワーカーとしてスケール。
次のボトルネックはPostgreSQLへのコネクション数 → RDS Proxyで吸収。

### Q6. サービス境界はどう決めた？なぜURLパスで分割しない？なぜ完全なモノリスでもない？
**A.** 境界は**URLパスではなく、トランザクション境界とデータ所有権**で決める。送金フロー・残高/台帳更新（DynamoDB）・送金メタ（PostgreSQL）はすべて1つの資金移動ユニットの一部で原子的に協調する必要があるため、URL（`/transfers`、`/transactions`、`/accounts`）で分割すると、コアの資金移動パスにメリットなく分散トランザクション/Sagaを強いることになる。よってこれらは両ストアを所有する単一の **Payments Service** にまとめる。逆に **Fraud**、**User Directory**、**Notification** は本質的に独立（データ・障害モード・スケールが別）なので、それらの障害が資金移動をブロックしないよう独立サービスとして存続させる。これは完全なモノリス（独立した関心事は分離している）でも、過剰分割されたメッシュ（原子的なコアはまとまっている）でもない。

### Q7. お金の額をどう表現する？浮動小数点は？
**A.** 使わない。`bigint` の minor unit（セント）で保持し、丸め誤差・表現誤差を排除する。通貨はUSD固定（要件）だが、将来の多通貨に備え通貨コードは保持する。

### Q8. 取消・返金はどう扱う？
**A.** 2つのケースがあり、いずれも追記専用（既存の `ledger_entries` は決して書き換えない）:
- **モラトリアム期間中の取消**（`SETTLED` 前）: 資金はまだエスクローにあるので、`REVERSAL` ペア（エスクロー → 送金者）を追記し `status=CANCELLED` にする。受取人は加算されていないので取り戻す対象はない。
- **決済後の返金**: 補償の `REVERSAL` ペア（受取人 → 送金者、エスクロー経由）を追記し `status=REVERSED` にする。

いずれも不変台帳が完全な監査証跡を保ち、`transfer_events` が変更のトリガー（誰が・何を）を記録する。
